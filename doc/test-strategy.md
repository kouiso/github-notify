# github-notify テスト戦略

## 1. 目的と対象

本ドキュメントは、Tauri デスクトップアプリ **github-notify** に対して、継続的に品質を担保するための包括的なテスト戦略を定義する。

- 対象スタック: React 19 + TypeScript + Vite + Tauri 2 + Rust + Tailwind CSS 4
- 既存テスト基盤: Vitest / Playwright / cargo test
- 既存 E2E シナリオ: `f01`〜`f11`

本戦略では、以下を明確化する。

1. テストピラミッド（unit / component / e2e / manual）
2. 機能ごとの正常系・異常系・エッジケース
3. 層ごとのカバレッジ目標
4. ツール責務分担（Vitest / Playwright / cargo test / 手動QA）
5. `e2e/scenarios/f01-f11` との対応表

---

## 2. テストピラミッド

### 2.1 レイヤー定義

| レイヤー | 主担当ツール | 主な対象 | 期待実行頻度 |
|---|---|---|---|
| Unit | Vitest / cargo test | 純粋関数、状態変換、ドメインロジック、Rustロジック | 毎PR |
| Component | Vitest + Testing Library | Reactコンポーネントの表示/操作/状態遷移 | 毎PR |
| E2E | Playwright (`e2e/scenarios`) | 画面遷移、主要ユーザーフロー、回帰検知 | 毎PR〜毎日 |
| Manual | 手動QAチェックリスト | ネイティブ連携、UX、OS依存挙動、探索的テスト | リリース前 |

### 2.2 ピラミッド方針

- 下層（Unit/Component）を厚くし、回帰検知の速度と安定性を優先。
- E2E は「重要フローの保証」に絞り、網羅は Unit/Component と分担。
- Tauri ネイティブ機能（トレイ、通知権限、ウィンドウ表示など）は手動QAを必須化。
- Rust 側のドメイン挙動（例: ポーリングや割り当て判定）は `cargo test` を第一責務とする。

---

## 3. カバレッジ目標

### 3.1 数値目標

| レイヤー | 指標 | 目標 |
|---|---|---|
| Frontend Unit/Component (Vitest) | lines/functions/branches/statements | 80%以上（既存閾値を維持） |
| Rust Unit (cargo test) | 主要モジュールの分岐網羅 | 重要ロジック100%の振る舞い観点（行数よりシナリオ重視） |
| E2E (Playwright) | 主要フロー | f01〜f11を常時実行可能状態 |
| Manual QA | リリース判定項目 | チェックリスト全項目実施、Fail 0 件 |

### 3.2 目標設定の補足

- Vitest の 80% 閾値は維持条件であり、重要機能追加時は目標引き上げを検討。
- E2E は flaky を避けるため、要素存在確認だけでなく「意味のある操作→結果」を増やす。
- 手動QAは自動化困難領域（通知、トレイ、起動時挙動、OS権限）を中心に運用。

## 5. 機能別テスト観点（正常系/異常系/エッジケース）

## 5.1 認証（Auth）

### 正常系
- Device Flow 開始でユーザーコード・検証URLが表示される。
- 認証完了後にアプリ本体画面へ遷移し、ポーリングが開始される。
- Personal Access Token 入力でログインできる。
- ログインキャンセルで初期画面に戻る。

### 異常系
- OAuth 開始API失敗時にエラーメッセージが表示される。
- 無効トークン入力時に拒否され、再入力可能である。
- 認証待機中タイムアウト時に再試行導線がある。

### エッジケース
- 連続クリック（多重開始）でも状態破綻しない。
- コピー操作（ユーザーコード）失敗時もアプリが停止しない。
- 既存トークン保持状態で起動した場合、不要なログイン画面を経由しない。

関連E2E: `f01-oauth-login`, `f11-onboarding`

## 5.2 Inbox（受信トレイ）

### 正常系
- 通知一覧が表示される（または空状態が適切に表示される）。
- アイテムクリックで該当 URL が開く。
- 既読化（単体/全件）が動作する。
- 手動更新で一覧が更新される。

### 異常系
- 取得API失敗時にエラー表示と再試行導線がある。
- 一部アイテムの既読化失敗時、失敗が認識できる。
- レスポンス遅延時にローディング状態が表示される。

### エッジケース
- 通知0件時の表示崩れがない。
- 非常に長いタイトル/リポジトリ名でも折返し・省略が適切。
- 大量件数でスクロール・選択操作が破綻しない。

関連E2E: `f02-inbox-display`, `f05-notification-click`, `f06-read-unread`

## 5.3 ダッシュボード（Dashboard）

### 正常系
- ダッシュボードタブ遷移が可能。
- レビュー対象PR、自分のPRセクションが表示される。
- リフレッシュで両セクションが更新される。

### 異常系
- 検索クエリ失敗時にセクション単位でエラー表示される。
- ユーザー情報未取得時でもクラッシュしない。

### エッジケース
- `@me` 置換対象がない場合でもクエリ生成が成立する。
- 組織フィルタ選択時、0件結果を正しく扱う。
- activeGroup と org 絞り込みの組合せで矛盾しない。

関連E2E: `f08-dashboard`

## 5.4 フィルター（通知フィルター）

### 正常系
- 設定画面でフィルター作成/編集/削除ができる。
- reason/repository 条件で通知絞り込みが反映される。
- global exclude reasons が全体に適用される。

### 異常系
- 不正なフィルター入力（空名など）が保存されない。
- 保存失敗時にUIが不整合にならない。

### エッジケース
- 条件なしフィルター（全件扱い）を期待通り解釈する。
- 複数フィルター重複時の件数集計が不自然にならない。
- 理由種別の追加/欠落があっても既存設定を安全に扱う。

関連E2E: `f04-filters`, `f09-settings`

## 5.5 グループ管理（Group Manager / RepositoryGroup）

### 正常系
- グループ作成（名前・色・対象リポジトリ）ができる。
- グループ編集/削除ができる。
- グループ単位の通知設定（デスクトップ通知/音/理由）を保存できる。

### 異常系
- 空名グループは保存不可。
- リポジトリ候補未取得時でもエラーにならない。

### エッジケース
- リポジトリ0件のグループを扱える。
- 100件超グループでもUI応答性を維持。
- 同名グループ作成時の運用ルール（許可/非許可）を明文化して検証。

関連E2E: なし (`f04-filters` はフィルター文言表示、`f09-settings` はダイアログ開閉とポーリング項目のみで、グループ作成/編集/削除は未カバー)

## 5.6 設定（Settings）

### 正常系
- 設定ダイアログの開閉ができる。
- ポーリング間隔・通知音・テーマ等の変更が反映される。
- 再起動後も設定が永続化される。

### 異常系
- 保存API失敗時にユーザーに失敗が伝わる。
- Keychain 状態取得失敗時も設定画面が利用可能。

### エッジケース
- 設定変更を短時間で連続操作しても最終状態が正しい。
- タブ切替中に保存処理が走っても破綻しない。
- 初回起動直後のデフォルト設定が期待値である。

関連E2E: `f09-settings`

## 5.7 通知（デスクトップ通知/通知クリック）

### 正常系
- 新着通知時にデスクトップ通知が発火する。
- クリックで対象PR/Issue/Discussion がブラウザで開く。
- 通知音ON時に音が再生される。

### 異常系
- OS通知権限拒否時にアプリがクラッシュしない。
- サウンド再生失敗時に処理全体が止まらない。

### エッジケース
- 連続通知時に過剰重複通知を抑制できる。
- 通知対象外reasonが混在するバッチで正しく選別される。
- グループ設定と全体設定の優先順位が一貫している。

関連E2E: なし (`f03-background-polling` はポーリングインジケータ表示、`f05-notification-click` はカーソルスタイル確認のみ。OS 通知/通知音/ブラウザ遷移は未カバー、Rust/ネイティブ層またはマニュアル QA に依存)

## 5.8 システムトレイ

### 正常系
- トレイメニューの 表示/非表示/終了 が機能する。
- トレイアイコンクリックでメインウィンドウが前面化する。

### 異常系
- アイコン取得失敗時に graceful degradation で起動継続する。
- ウィンドウ参照失敗時にクラッシュせずログ記録される。

### エッジケース
- アプリ最小化状態からの再表示がOS差分なく機能する。
- 複数ディスプレイ環境で復帰位置が異常にならない。
- 高DPI環境でアイコン表示が崩れない。

関連E2E: `f10-system-tray`（WebViewではskip、手動必須）

## 5.9 オンボーディング

### 正常系
- 初回起動でオンボーディングが表示される。
- 完了操作で `onboardingCompleted` が保存される。
- 2回目起動以降は不要表示されない。

### 異常系
- 設定保存失敗時に完了状態が誤って確定しない。
- ダイアログ閉じる操作でも完了処理が期待通りに行われる。

### エッジケース
- 途中でアプリ再起動しても整合した表示状態を維持。
- 画面サイズが小さい環境でも操作ボタンが到達可能。
- 多言語表示切替（将来対応）時の文言崩れに備える。

関連E2E: `f11-onboarding`, `f01-oauth-login`

## 6. E2Eシナリオ対応マップ（f01〜f11）

| シナリオ | 主機能 | 補足 | 手動QA要否 |
|---|---|---|---|
| f01 oauth-login | 認証 | Device Flow開始・キャンセル | 要（実OAuth障害系） |
| f02 inbox-display | Inbox表示 | 一覧/空状態 | 要（大量データ・視認性） |
| f03 background-polling | バックグラウンド | インジケータ中心 | 要（OS通知/音） |
| f04 filters | フィルター | 設定画面の存在確認中心 | 要（実絞り込み精査） |
| f05 notification-click | 通知クリック | `cursor: pointer` の style 確認のみ (クリック実行/shell.open 呼び出しは未検証) | 必須（実ブラウザ遷移とクリック→URL展開を手動で確認） |
| f06 read-unread | 既読/未読 | Mark all存在確認中心 | 要（同期整合） |
| f07 unassign-detection | 担当外検出 | E2E skip、Rust側重視 | 要（UIラベル確認） |
| f08 dashboard | ダッシュボード | タブ遷移・セクション表示 | 要（実データ妥当性） |
| f09 settings | 設定 | 開閉と一部設定項目 | 要（永続化/失敗系） |
| f10 system-tray | システムトレイ | E2E skip | 必須（ネイティブ検証） |
| f11 onboarding | 初回導線 | 初回表示・歓迎導線 | 要（初回状態再現） |

---

## 7. リリースゲート（推奨）

1. Vitest 緑（閾値維持）
2. cargo test 緑
3. Playwright f01〜f11 実行（skip理由を記録）
4. `doc/manual-qa-checklist.md` の手動項目を実施
5. Fail 0件、または既知問題としてリリースノートに明示

---

## 8. 継続的改善ポイント

- f04/f06 は現在「存在確認」が中心のため、状態変化検証を追加する。
- f07/f10 のようなネイティブ依存領域は、将来的にTauri統合テスト導入を検討する。
- Flaky 観測時は retry に依存せず、待機条件とテストデータ制御を先に見直す。
- 手動QAで頻出する不具合は優先的に自動化候補へ昇格する。
